# 이 블로그에 대해

Hugo로 만든 애드센스 수익화 목적 블로그. 키워드는 검색량이 0은 아니면서, 경쟁은 약하고, 단가는 받쳐 주는' 키워드.주제는 실시간으로 화제가 되는 주제. 콘텐츠는 정보성이되, 실행형/비교형 키워드(신청 방법, 비교,
조건 등)를 우선한다.

## 새 글 작성 시 반드시 지킬 구조

### Front matter

```yaml
---
title: "핵심 키워드를 앞에 둔 제목"
date: YYYY-MM-DD
draft: true   # 검수 끝나면 false로 변경 후 발행
categories: ["대분류"]   # 사이트 상단 카테고리 클러스터 (예: 법률, 세금, 재테크, 정책)
tags: ["롱테일 키워드1", "롱테일 키워드2"]   # 본문에서 실제로 다룬 구체적 롱테일
description: "검색결과 미리보기 문장. 클릭하면 얻는 이득을 압축해서 쓴다."
key_points:   # 글 상단 "핵심 요약" 박스에 그대로 노출되는 불릿 2~4개 (front matter 예약어인
              # `summary`와 겹치면 Hugo가 자동 요약과 충돌하므로 반드시 이 이름을 쓴다)
  - "신청 대상: ..."
  - "지원 금액: ..."
  - "신청 방법: ..."
  - "신청 기한: ..."
---
```

`key_points`는 두괄식 서론 문단을 재활용하는 게 아니라 완전히 별도로 짧게 쓴다. 각 불릿은
"라벨: 핵심 값" 형태로 한 줄에 사실 하나만 담는다 — 신청 방법 글이면 신청 대상/지원 금액/
신청 방법/신청 기한처럼, 독자가 스크롤 없이 바로 확인하고 싶어 할 항목 위주로 2~4개만
고른다. 서론 문단처럼 여러 조건을 한 문장에 욱여넣지 말 것.

### 글의 구조

- 제목: 핵심 키워드를 앞쪽에 둔다. 검색 결과에서 클릭을 부르되 낚시는 금물. 낚시 제목은 단기 클릭을 얻을지 몰라도, 들어온 사람이 곧장 실망해 떠나면 그 이탈이 순위를 깎는다. 결국 손해다.

- 서론: 두괄식. 첫 문단에서 독자의 질문에 곧장 답한다. 검색해서 들어온 사람은 결론부터 원한다. 빙빙 돌리면 바로 뒤로 가기를 누른다. "~에 대해 알아보겠습니다" 같은 예고형 서론은 최악이다. 알아보겠다고 예고만 하고 정작 답을 미루면 독자는 그 자리에서 떠난다.

- 본문: 소제목으로 논리 단위를 나눈다. 검색 엔진은 소제목으로 글의 구조와 주제 범위를 읽는다. 각 소제목 아래엔 구체적 정보·단계·수치를 담는다. 소제목 자체도 독자가 스캔하다 멈출 만큼 구체적이어야 한다. "장점"보다 "배당락 전에 사면 손해 보는 경우"가 낫다.

- FAQ: 글 끝에 자주 묻는 질문 3~5개. 이건 검색 결과의 질문-답변 영역에 노출될 확률을 높이고, 롱테일 질문을 추가로 흡수한다. 질문에 답변을 나중에 포스팅하겠다 라고 해도됨 그리고 기록 나중에 컨텐츠로 활용하고 최종적으로 컨텐츠 추가되면 링크로 연결
FAQ를 '구색 맞추기'로 여기면 안 된다. 잘 짜인 FAQ의 각 질문은 그 자체로 또 하나의 롱테일 키워드다. 본문은 '배당락일 계산법'을 다루더라도, FAQ에 "배당받고 바로 팔아도 되나요?"를 넣으면 그 질문을 검색한 사람까지 흡수한다. 한 편의 글로 본문 키워드 하나가 아니라 질문 서너 개까지 그물에 거는 것 — FAQ는 그 그물이다. 그러니 프롬프트에서 FAQ를 '글에 실제로 나온 내용을 기반으로, 사람들이 진짜 궁금해할 질문'으로 만들라고 지정해야지, 막연히 "FAQ 넣어 줘"라고만 하면 뻔한 자문자답이 나온다.

### 본문

1. **서론은 두괄식**: 첫 문단에서 바로 답한다. "~에 대해 알아보겠습니다" 같은 예고형 서론 금지.
2. **소제목은 구체적으로**: "장점"이 아니라 "OO인 경우 손해 보는 이유"처럼 스캔하다 멈출 만큼
   구체적으로 쓴다. 각 소제목 아래엔 수치·단계·조건 등 실질 정보를 담는다.
3. **공식 신청/신청 방법 관련 글이면 실제 신청 링크를 반드시 포함**한다.
4. **FAQ 3~5개를 글 끝에 배치**한다. 본문에서 다룬 내용을 기반으로 하되, 본문 제목과는 다른
   각도의 진짜 롱테일 질문으로 구성한다 (본문이 다루지 않은 인접 질문을 흡수하는 그물 역할).
5. **YMYL(법률/세금/의료/금융) 주제는 팩트체크 필수**. 확실하지 않은 수치·조건은 웹 검색으로
   최신 정보를 확인하고, 글 하단에 "정보 제공 목적이며 정확한 확인은 공식 기관에서" 식의
   면책 문구를 넣는다.

### 글 유형 다양화: 신청방법형 vs 비교형

지금까지 쓴 글이 전부 "신청방법형"(두괄식 서론 → 핵심요약 박스 → 대상/금액/방법/기한 소제목
→ FAQ) 한 가지 틀만 반복해서, 사람이 보면 판박이로 찍어낸 것처럼 보인다는 지적이 있었다.
앞으로는 주제 성격에 따라 아래 두 유형을 실제로 구분해서 쓴다.

- **신청방법형** (기존 방식): 지원금·급여·바우처처럼 "신청"이 핵심인 주제. 대상/금액/방법/
  기한 위주로 구성한다.
- **비교형** (신규): "OO vs OO", "작년 대비 달라진 점", "이 제도 저 제도 비교"처럼 두 가지
  이상을 견주는 게 핵심인 주제. 이런 글은:
  - 서론에서 바로 결론(핵심 차이 한 줄)부터 제시한다.
  - 본문에 마크다운 표를 최소 1개 이상 써서 항목별로 나란히 비교한다 (예: `구분 | 기존 | 변경 후`).
  - 소제목은 "OO 항목에서 이렇게 달라졌다"처럼 차이점 중심으로 잡는다.
  - `key_points`도 "무엇이 달라졌는지" 위주 불릿으로 채운다.

주제 후보를 고를 때 실행형(신청 방법)과 비교형(비교, 달라진 점, vs) 키워드를 매번 기계적으로
같은 쪽만 고르지 말고 번갈아 섞는다.

**표 활용**: 가구원수별 금액, 기존/변경 비교처럼 숫자·조건이 여러 줄 나열되는 구간은 불릿
목록만 쓰지 말고 마크다운 표를 우선 고려한다.

**서론 문장 패턴 다양화**: "만 OO세가 OO하면 최대 OO원" 공식을 매번 반복하지 않는다.
질문형 서론("OO 신청, 자격 되는지 헷갈린다면?"), 결론 선언형("OO는 이제 신청 안 해도
됩니다"), 비교형("OO와 OO의 차이는 딱 하나다") 등으로 글마다 다르게 시작한다.

### 카테고리/태그 체계

- `categories`: 큰 주제 클러스터 (사이트 상단 "카테고리" 메뉴에서 모아보기, `/categories/`)
- `tags`: 세부 롱테일 키워드 (`/tags/`)
- 현재 사용 중인 카테고리: 법률, 세금, 재테크, 정책. 새 글을 쓸 때 이 중 어울리는 게 있으면
  그걸 쓰고, 없으면 새 카테고리를 만들어 쓴다 (만든 카테고리는 이 목록에 추가해 갱신한다).
  카테고리 개수가 너무 잘게 쪼개지지 않도록, 새로 만들기 전에 기존 카테고리로 커버되는지
  한 번 더 검토한다.

## 네이버 블로그용 버전 함께 만들기

GitHub 블로그 글(`content/posts/`)을 발행할 때마다, 같은 내용을 네이버 블로그에 그대로
복사해 붙여넣을 수 있는 버전도 함께 만든다. 이 버전은 Hugo 사이트에는 배포되지 않고,
네이버 블로그 에디터에 수동으로 붙여넣기 위한 텍스트다.

### 저장 위치

`naver-posts/<원본 글과 동일한 파일명>.md`에 저장한다 (`content/` 밖이라 Hugo 빌드에는
포함되지 않는다). GitHub 글과 마찬가지로 커밋에 포함시키고, 채팅 응답에도 전체 내용을
그대로 출력해서 바로 복사해 쓸 수 있게 한다.

### 제목: 시선을 사로잡는 제목으로 별도로 뽑는다

GitHub 버전 제목은 SEO 키워드를 앞에 두는 담백한 제목이지만, 네이버 버전 제목은 클릭을
부르는 캐치한 제목으로 새로 짓는다. 아래 패턴을 번갈아 활용한다.

- **공감 질문형**: "OO 때문에 고민이셨다면?"
- **손해 강조형**: "이거 모르면 OO원 손해 봅니다"
- **숫자 임팩트형**: "매달 20만원씩 24개월, 총 480만원 받는 법"
- **반전/비교형**: "예전엔 안 됐는데 이제는 됩니다"

과장이나 거짓으로 낚는 것과는 다르다 — 실제 내용과 어긋나지 않는 선에서 호기심과 숫자로
시선을 끄는 것이 핵심이다.

### 어조: 친근하고 따뜻하게

- GitHub 버전의 담백한 정보체(~다, ~한다) 대신, **친근한 존댓말**(~해요, ~예요, ~랍니다)로
  다시 쓴다.
- 옆에서 챙겨주듯 말을 건다. "이런 분들 꼭 확인하세요!", "여기서 포인트!"처럼 대화하듯
  이어간다.
- 글 시작에 짧은 인사말, 글 끝에 마무리 인사(예: "오늘 정보가 도움 되셨길 바라요 :)")를
  붙인다.

### 시각적으로 예쁘게

- 소제목마다 어울리는 이모지를 하나씩 붙인다 (예: "📌 신청 대상은 이래요", "💰 얼마나 받을
  수 있을까요?").
- 문단은 2~3문장 이내로 짧게 끊는다.
- 핵심 수치나 기한은 **굵게** 표시하고, 구분선(─────)이나 이모지 불릿(✅, 👉, 💡)을
  섹션 사이사이에 써서 리듬을 준다.

### 내용/구조

- 핵심 사실(대상, 금액, 방법, 기한)은 GitHub 버전과 동일하게 유지한다 — 이미 팩트체크된
  내용이므로 다시 검증할 필요는 없다.
- 외부 링크는 최소화한다 (네이버 알고리즘은 외부 링크가 많은 글을 덜 선호하는 경향이 있다).
  공식 신청처는 "OOO에서 검색해보세요" 식으로 링크 없이 안내해도 된다.
- FAQ는 유지하되, 질문·답변도 같은 친근한 톤으로 다시 쓴다.

## Git 작업 방침

- 작업 브랜치(main이 아닌 feature/claude 브랜치)에 대한 commit과 push는 매번 확인받지 않고 바로 진행한다.
  최종 검토와 main 머지는 항상 사람이 직접 한다.
- 작업 브랜치를 push한 뒤에는 확인받지 않고 바로 main 대상 PR을 생성한다.
- main으로의 머지, force push 등 main에 직접 영향을 주는 작업은 반드시 먼저 확인받는다.
